What it refuses to claim

the honesty is the product

The Amdahl ceiling is always on screen

A fix in fraction f of the runtime can never beat 1/(1−f). The measured bar stops at that wall by construction — illustratively, a 700× kernel inside 40% of runtime is capped at the 1.67× it really delivers (1/(0.6 + 0.4/700)), never 700×.

Kernel ≠ whole-program

Every ratio is the warmup-aware whole-program wall-clock, measured against a neutral baseline. We never put a kernel number where a whole-program number belongs.

The grade is output confidence

exact / probabilistic / decline is decided at runtime by the input and the verifier — not a fixed property of a fixer. The same rewrite can be exact on integers and decline on floats.

Component speedups don't multiply

Illustratively, 3× · 20× · 6.7× component wins do not compound to 400×. The cumulative number is never a product of component factors — it is re-measured fresh, end to end, every round.

Big wins are the minority — measured, not promised

On a production-representative corpus most code is I/O and control flow with no foldable asymptotic structure, so only a low-single-digit fraction folds to a big win — the rest is a modest constant-factor or an honest decline. The 115× above is real but selected; the engine's value is telling you the truth either way, not promising a uniform speedup.
